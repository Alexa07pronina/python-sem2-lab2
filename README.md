# Лабораторная работа №2. Модель задачи: дескрипторы и @property


## Цель работы
Освоить управление доступом к атрибутам и защиту инвариантов доменной
модели.
## Постановка задачи
Реализовать модель задачи ( Task ) в рамках платформы обработки задач с
корректной инкапсуляцией и валидацией состояния.

## Атрибуты задачи
- идентификатор задачи;
- описание задачи;
- приоритет;
- статус задачи;
- время создания;
- вычисляемые свойства (готовность к выполнению, возраст задачи);

### Примеры источников задач:
- задачи, загружаемые из файла;
- задачи, генерируемые программно (генератор);
- задачи, получаемые из API-заглушки; (под API-заглушкой понимается упрощённый
программный компонент, имитирующий внешний источник задач)


## Вывод
В ходе выполнения работы была реализована модель задачи (Task) с корректной инкапсуляцией и валидацией состояния. 
Использование пользовательских дескрипторов (PriorityDescriptor, StatusDescriptor) для валидации атрибутов при записи обеспечивает защиту инвариантов объекта: приоритет всегда находится в диапазоне 1–5, а статус принимает только допустимые строковые значения. 
Также продемонстрировано различие между data-дескрипторами (имеют set, перехватывают запись) и non-data-дескрипторами (@property без setter, read-only). Важно отметить, что slots не только экономит память, но и предотвращает создание случайных атрибутов, усиливая инкапсуляцию.
Архитектурно модель разделена на публичный API (дескрипторы и property) и внутреннее состояние (защищённые поля с префиксом _). Идентификатор генерируется сервисом автоматически (UUID), что исключает дублирование и ошибки пользователя. 
Статус реализован строкой для удобства API, а методы start(), complete(), fail(), retry() обеспечивают контролируемые переходы между состояниями.
